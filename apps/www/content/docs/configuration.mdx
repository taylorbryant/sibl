# Configuration

The config is both a runtime-validated object and the source of the public navigation hierarchy.

## Project fields

| Field | Default | Purpose |
| --- | --- | --- |
| `title` | required | Product or project name |
| `description` | required | Site and agent-output summary |
| `basePath` | `/docs` | Route prefix for documentation |
| `deploymentBasePath` | `""` | Optional path where the entire Next application is deployed |
| `siteUrl` | — | Optional origin for canonical agent links |
| `links` | `[]` | Project links shown beside the sidebar controls |
| `outputs` | standard paths | Locations for search and agent outputs |
| `search` | enabled | Search visibility and placeholder |
| `theme` | Sibl defaults | Light/dark backgrounds and accents, mark, and storage key |

## Navigation sections

Each section has a label and one or more page entries. A page entry maps its public slug to an explicit `.md` or `.mdx` source.

```ts
{
  label: "Reference",
  items: [
    {
      title: "Configuration",
      navLabel: "Config",
      slug: "configuration",
      source: "content/docs/configuration.mdx",
      description: "Every supported configuration field.",
      eyebrow: "Reference",
      badge: "API",
    },
  ],
}
```

Slugs use lowercase URL segments such as `guides/deployment`. The root page uses an empty slug. Source paths must stay inside the application root.

## Deployment paths

`basePath` and `deploymentBasePath` solve different problems. `basePath` places documentation inside the application, such as `/docs`. `deploymentBasePath` mirrors Next.js `basePath` when the whole application is hosted below a path, such as `/preview`.

```ts
const deploymentBasePath = (process.env.NEXT_PUBLIC_BASE_PATH ?? "").replace(
  /\/+$/,
  "",
);

export default defineDocs({
  // ...
  basePath: "/docs",
  deploymentBasePath,
});
```

Pass the same value to `basePath` in `next.config.ts`. Sibl applies it to public search and agent-output URLs while leaving Next.js to prefix router links.

## Browser chrome

`theme.background` and `theme.backgroundDark` color the documentation canvas and provide the values for browser chrome on supported browsers. Export them through Next.js viewport metadata, then pass them to the pre-hydration theme script:

```tsx
export const viewport = {
  themeColor: [
    { color: config.theme.background, media: "(prefers-color-scheme: light)" },
    { color: config.theme.backgroundDark, media: "(prefers-color-scheme: dark)" },
  ],
};

<DocsThemeScript
  darkColor={config.theme.backgroundDark}
  lightColor={config.theme.background}
  storageKey={config.theme.storageKey}
/>;
```

The media-aware metadata supplies the system default. `DocsThemeScript` and the built-in toggle update `theme-color` when the stored Sibl preference differs from the device setting.

## Code highlighting

`createMdxOptions` uses `github-light` and `github-dark` by default. Pass any bundled Shiki theme names to choose a different pair:

```ts
const withMDX = createMDX({
  options: createMdxOptions({
    themes: {
      light: "vitesse-light",
      dark: "vitesse-dark",
    },
  }),
});
```

## Composition settings

UI customization is passed as React props rather than stored in the serializable manifest:

```tsx
<DocsPage
  config={docs.config}
  page={page}
  brand={<ProductLogo />}
  headerActions={<StatusLink />}
  sidebarFooter={<Version />}
  footer={<LegalLinks />}
>
  <Content />
</DocsPage>
```

`DocsPage` expects the MDX file to own its `# Title`. Use the explicit `DocsPageWithHeader` composition when the manifest should render the page heading instead.
