# Architecture

Sibl separates the documentation problem into three layers. Applications can adopt the complete default or compose the individual pieces themselves.

## The typed manifest

`defineDocs` validates project metadata, navigation, slugs, source files, output paths, links, search settings, and theme tokens. Duplicate routes or sources fail immediately.

The manifest is deliberately more explicit than filesystem-derived navigation. Moving an MDX file should not silently reorganize a public information architecture.

## Next-compiled MDX

Next.js imports and compiles every MDX module. Sibl never evaluates MDX source at request time. A small content registry connects the manifest slug to the statically imported component:

```ts
import Overview from "@/content/docs/index.mdx";
import Installation from "@/content/docs/installation.mdx";
import { docs } from "@/lib/docs";

export const content = docs.defineContent({
  "": Overview,
  installation: Installation,
});
```

Sibl reads the raw source only when validating content or producing search and agent outputs. `docs.defineContent()` fails when the registry and navigation differ, while `docs.validate()` reports missing source files, documentation pages, and heading anchors.

## The composable React surface

`DocsPage` combines the default Sibl shell with an application-owned MDX title. `DocsPageWithHeader` is an explicit alternative for sites that want to derive the title and description from the manifest. The same page can also be assembled directly from `DocsLayout`, `DocsArticle`, `DocsPageHeader`, and `DocsPagination`.

The shell accepts application-owned branding, actions, sidebar metadata, and footer content. Product palettes can keep using semantic CSS variables such as `--color-bg`, `--color-accent`, and `--color-ink`. `createMdxComponents` also accepts overrides for every rendered element.

## Small client islands

MDX content remains server-rendered. The shell uses small client islands only for behavior that needs the browser: mobile navigation, keyboard search, theme persistence, table-of-contents tracking, and code copying.

## The framework boundary

Sibl does not wrap `next dev`, hide routes, generate an application, or invent a deployment pipeline. This is an architectural constraint, not a missing feature.
